# 렉시컬 환경 - 실행컨텍스트

## 실행컨텍스트 실행 과정

1. 코드가 실행되면 JS는 전체스캔
2. 콜스택이라는 통에 전역 실컨 담음
3. 그위에 또 함수나 실컨 담음
4. 위에서부터 실행되고 함수가 모두 실행되고 나면
   전역 실컨 종료
5. 모두 사라짐(콜스텍내에서)

***



## 호이스팅 이해하기

함수가 선언된 이후 변수 로그 => 변수 내용 실행
선언되기 전 실행 => undefined (에러가 나지 않음)
끌어올려진 듯한 현상 == 호이스팅;

why
: JS엔진이 코드스캔할때 변수같은 것 을 RECORD 에 기록해두기 때문.

### RECRD 

== 식별자와 식별자에 바인딩된 값을 기록해두는 객체

호이스팅은
변수 H / 함수 H로 나뉨

### 변수H

(생성단계)

1. 코드실행하면, 우선 전역실컨 생성해 콜스택에 담음
2. 전체 스캔 후 선언할 게 있으면 선언
3. var의 경우 전역실컨 REC에 저장 후 undefined로 값 초기화해둠

(실행단계)

1. 스캔하면서 undefined로 초기화 해둔 값에 변수내용 업데이트 및 실행

2. let const로 실행한 변수라면? 선언 전 참조가 힘듦(일시적 사각지대)

   즉 위 생성단계의 undefined로 초기화가 안되므로 선언전까지 식별자 참조 불가

### 함수H

함수 표현식의 경우 함수를 변수에 담기때문에 변수H와 동일하게 동작

선언문 방식은 선언과 동시에 함수가 생성되어 어디서든 선언가능

### Outer정체

: 외부 환경 참조

식별자를 결정할 때 현재 실컨과 다른 실컨을 연결하여
현재 실컨 외의 변수/함수 들을 찾을 수 있게 만든 것

식별자 결정은 예를들면 전역실컨  => Outer => 함수 => C => 변수
를 찾은 후 실행. 찾으면 종료
못찾으면,  Outer로 나간 통로를 다시 들어와 전역까지 와서 찾음

이 때 이러한 스코프들의 연결리스트를

### 스코프체인

이라고 하고 그 과정을 

### 스코프체이닝 

이라고 함
