# Resource
- 리소스의 의미?
    - 회원을 등록하고 수정하고 조회하는 것은 리소스가 아니다.
    - ex) 회원 수정 -> 회원이 리소스
    - **회원이라는 개념 자체가 리소스다.**

<br>

# HTTP 메서드 종류
## 주요 메서드
- GET: 리소스 조회
- POST: 요청 데이터 처리, 주로 등록에 사용
- PUT: 리소스를 대체, 해당 리소스가 없으면 생성
- PATCH: 리소스 부분 변경
- DELETE: 리소스 삭제

## 기타 메서드
- HEAD: GET과 동일하지만 메시지 부분을 제외하고, 상태 줄과 헤더만 변환
- OPTIONS: 대상 리소스에 대한 통신 가능 메서드를 설명(주로 CORS에서 사용)

<br>

# HTTP 메서드 - GET, POST
## GET
```http
GET /search?q=hello&hl=ko HTTP/1.1
Host: www.google.com
```
- 리소스 조회
- 서버에 전달하고 싶은 데이터는 query(쿼리 파라미터, 쿼리 스트링)를 통해서 전달
- 메시지 바디를 사용해서 데이터를 전달할 수 있지만, 지원하지 않는 곳이 많아서 **권장하지 않음**

### 리소스 조회1 - 메시지 전달
**[클라이언트]**
```http
GET /members/100 HTTP/1.1
Host: localhost:8080
```
- GET 메서드를 통해 /members/100 리소스를 조회

**[서버]**
`/members/100`
```http
{
    "username": "young",
    "age: 20
}
```
---

### 리소스 조회2 - 서버 도착 후 응답 데이터
**[서버]**
`/members/100`
```http
{
    "username": "young",
    "age: 20
}
```
- Response로 클라이언트에 응답 데이터 전송

**클라이언트 응답 데이터**
```http
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 34

{
    "username": "young",
    "age: 20 
}
```

## POST
```http
POST /members HTTP/1.1
Content-Type: application/json

{
    "username": "young",
    "age: 20
}
```

- 요청 데이터 처리
- **메시지 바디를 통해 서버로 요청 데이터 전달**
- 서버는 요청 데이터를 **처리**
    - 메시지 바디를 통해 들어온 데이터를 처리하는 모든 기능을 수행
- 주로 전달된 테이터로 신규 리소스 등록, 프로세스 처리에 사용

### 리소스 등록1 - 메시지 전달
**[클라이언트]**
```http
POST /members HTTP/1.1
Content-Type: application/json

{
    "username": "young",
    "age: 20
}
```
- POST 메서드를 사용해 서버측에 메시지를 전달
- 서버측에는 `/members`외에 아무것도 없는 상황

---

### 리소스 등록2 - 신규 리소스 생성
**[서버]**
`/members` -> `/members/100` (신규 리소스 식별자 생성)
```http
{
    "username": "young",
    "age: 20
}
```
- 클라이언트는 등록될 리소스의 URI를 모른다.
- 서버가 새로 등록된 리소스 URI를 자동으로 생성해준다.

---

### 리소스 등록3 - 응답 데이터
**클라이언트 응답 데이터**
```http
HTTP/1.1 201 OK
Content-Type: application/json
Content-Length: 34
Location: /members/100

{
    "username": "young",
    "age: 20 
}
```

## POST 사용 예시
- HTML 양식에 입력된 필드와 같은 데이터 블록을 데이터 처리 프로세스에 제공
    - ex) HTML FORM에 입력한 정보로 회원 가입, 주문 등에서 사용
- 게시판, 뉴스 그룹, 메일링 리스트, 블로그 등 메시지 게시
    - ex) 게시판 글쓰기, 댓글 달기
- 서버가 아직 식별하지 않는 새 리소스 생성
    - ex) 신규 주문 생성
- 기존 자원에 데이터 추가
    - ex) 한 문서 끝에 내용 추가하기
**이 리소스 URI에 POST 요청이 오면 요청 데이터를 어떻게 처리할지 리소스마다 따로 정해야 함 -> 정해진 것이 없음**

## POST 정리
- **1. 새 리소스 생성(등록)**
    - 서버가 아직 식별하지 않는 새 리소스 생성
- **2. 요청 데이터 처리**
    - 단순히 데이터를 생성하거나, 변경하는 것을 넘어서 프로세스를 처리해야 하는 경우
    - ex) 주문에서 결제완료 -> 배달시작 -> 배달완료 처럼 단순히 값 변경을 넘어 프로세스의 상태가 변경되는 경우
    - POST의 결과로 새로운 리소스가 생성되지 않을 수도 있음
    - ex) POST /orders/{orderId}/start-delivery (**컨트롤 URI**)
- **3. 다른 메서드로 처리하기 애매한 경우**
    - ex) JSON으로 조회 데이터를 넘겨야 하는데, GET 메서드를 사용하기 어려운 경우
    - 애매하면 POST
